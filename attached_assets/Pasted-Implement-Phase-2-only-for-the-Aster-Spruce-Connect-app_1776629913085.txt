Updated product direction:
The Progress Timeline is now the primary organizing experience in the app.
The app should feel centered around project progress, milestones, updates, next steps, and client clarity.

Planning Board remains in the product, but it is NOT a core feature.
It should be treated as a supporting collaboration/design tool, not the main organizing concept.

Crew workflow remains unchanged outside the app.
Do not build out new crew portal features in this phase.
Do not reintroduce payroll or SMS concepts.

Phase 2 goal:
Refactor and improve Dashboard and ProjectDetails so that the Progress Timeline becomes the clearest and most important experience for both internal users and clients.

This phase is about:
- making the timeline the hero feature
- improving project clarity and wayfinding
- splitting large UI files into maintainable components
- making ProjectDetails feel like a true project progress hub
- making Dashboard point users toward current status, next actions, and project movement

This phase is NOT about:
- backend route splitting
- schema changes
- destructive refactors
- changing existing permissions
- replacing the Planning Board
- building new crew workflows
- rewriting the app from scratch

Critical rules:
- Preserve all existing populated data.
- Do not wipe, reset, or overwrite the database.
- Do not remove or rename schema fields.
- Do not change existing API endpoints unless absolutely necessary for existing UI wiring.
- Do not weaken role-based access control.
- Do not rewrite the app from scratch.
- Make additive, incremental, non-destructive refactors only.
- Preserve current functionality while improving structure, hierarchy, and focus. Incremental refactoring in small steps is safer than broad rewrites. [web:1166][web:999]

Primary UX objective:
Users should quickly understand:
- where the project is now
- what has been completed
- what is currently happening
- what is coming next
- what changed
- whether they need to take action

The app should feel less like a collection of tools and more like a project journey centered on visible progress and next steps. Journey-centric and client-centric design emphasize guiding users through the process rather than making them hunt through unrelated sections. [web:1159][web:1167][web:628]

Success criteria:
- Dashboard is reorganized to emphasize project progress and next actions
- ProjectDetails is reorganized around a strong Progress Timeline / project journey model
- Timeline-related information is surfaced before secondary tools
- Planning Board is still accessible but clearly secondary
- Client users see an especially clear, reassuring, progress-first experience
- Existing logic, data, routes, and permissions continue to work
- Code becomes easier to maintain through smaller extracted components

Implementation plan:

1. Refactor Dashboard into role-specific timeline-aware views
Take the current Dashboard.tsx and split it into smaller components such as:
- AdminDashboardView
- CrewDashboardView
- ClientDashboardView

Keep Dashboard.tsx as the page/container only.

Dashboard container responsibilities:
- auth + effective role resolution
- top-level loading and error states
- shared page shell and header
- pass required data into role-specific components

Role-specific intent:

AdminDashboardView:
- show project progress summary first
- highlight projects needing attention
- highlight timeline risk, overdue items, stalled milestones, or pending decisions if already supported by data
- surface quick links into ProjectDetails and timeline-related sections

CrewDashboardView:
- keep lightweight and action-oriented
- surface today / next work items and schedule context only if already supported
- do not expand this into a large crew feature area

ClientDashboardView:
- make this especially calm and reassuring
- show current project stage
- next milestone
- latest meaningful update
- any pending client decision or requested action
- direct path into their project’s progress timeline

Do not invent new workflows.
Reorganize existing content to make progress and next steps more obvious.

2. Make Dashboard feel progress-first
For each role view:
- place progress status near the top
- show the most important “what’s happening now” information before secondary modules
- reduce equal-weight sections
- make “next action” and “latest update” visually obvious
- ensure clients can understand their project without exploring many tabs

A good client portal should guide users toward what needs attention immediately instead of making them search through navigation or dense screens. [web:628][web:1160]

3. Refactor ProjectDetails into a true project progress hub
Take the large ProjectDetails page and split it into smaller components.

Suggested structure:
- ProjectDetails.tsx becomes the shell/container
- ProjectHeader
- ProjectProgressSummary
- ProjectTimelineSection
- ProjectNextStepsCard or PendingActionsCard
- supporting tabs/sections for secondary tools

You may also extract supporting sections such as:
- ProjectUpdatesSection
- ProjectDocumentsSection
- ProjectPhotosSection
- ProjectPlanningSection
- ProjectFinancialSection

Preserve the existing functional behavior, but reorganize the page so progress and timeline context come first.

4. Make the Progress Timeline the hero section in ProjectDetails
This is the most important requirement in this phase.

ProjectDetails should prioritize:
- current stage / phase
- completed milestones
- active/current milestone
- upcoming milestone(s)
- latest update
- pending client/admin decision
- linked supporting context such as documents, photos, or notes if already present

The timeline should not feel buried.
It should be the clearest and most prominent organizing section of the project page.

Timeline design guidance:
- focus on actions/events with dates
- make completed/current/upcoming states easy to distinguish
- allow supporting context to hang off timeline events when available
- make “what happens next” obvious
Timelines are most useful when centered on user-relevant actions and dated milestones rather than abstract status alone. [web:1164][web:1153]

5. Reposition Planning Board as secondary
Keep Planning Board in the product, but make it visibly secondary to project progress.

Guidance:
- do not remove it
- do not expand it
- do not treat it as the primary tab or hero section
- if it appears in ProjectDetails, place it after progress-first content
- if needed, relabel or reposition it as a supporting planning/collaboration area

6. Improve client-facing clarity in ProjectDetails
For client users:
- reduce internal/admin noise
- use simpler language where feasible
- surface progress, next step, updates, and decisions first
- make the experience feel reassuring and premium
- avoid overwhelming the client with too many equal-priority modules

7. Improve component boundaries and maintainability
As you refactor:
- keep data fetching logic in page containers or hooks where practical
- keep presentational components focused on UI
- reduce large inline render blocks
- extract components with clear names and focused responsibilities
- prefer small clean refactors over risky redesigns

8. Preserve adaptive behavior from Phase 1
Do not regress:
- desktop/sidebar navigation
- mobile usability
- compact ProjectDetails mobile behavior
- Planning Board phone limitations
- banners or expectations for dense tools on phone if already implemented

9. Validation and QA
Before finishing, verify:
- Dashboard still works for admin, crew, and client
- role switching still works for admin preview mode
- ProjectDetails still loads all expected content
- timeline/progress information is visually primary
- Planning Board remains accessible but secondary
- client users only see allowed information
- mobile and desktop layouts both still work

- no populated data has changed or been removed
