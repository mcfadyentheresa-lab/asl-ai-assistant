Implement Phase 2 only for the Aster & Spruce Connect app.

Phase 2 goal:
Refactor and improve the two most important user-facing hubs:
1. Dashboard
2. Project Details

This phase is about:
- clearer information hierarchy
- role-specific presentation
- smaller reusable components
- better readability and maintainability
- improved client-facing polish

This phase is NOT about:
- backend route splitting
- schema changes
- destructive refactors
- new workflows
- changing role guards
- rewriting the full Planning Board engine
- redesigning the entire visual brand

Critical rules:
- Preserve all existing populated data.
- Do not wipe, reset, or overwrite the database.
- Do not remove or rename schema fields.
- Do not change existing API endpoints unless absolutely necessary for existing UI wiring.
- Do not weaken role-based access control.
- Do not rewrite the app from scratch.
- Make additive, incremental, non-destructive refactors only.
- Preserve current functionality while improving structure and layout. Incremental refactoring is preferred over broad rewrites. [web:1054][web:1060]

Primary UX problem to solve:
The Dashboard currently contains three substantially different role views in one large file, and ProjectDetails behaves like a large multi-tab container with heavy inline content. This reduces clarity for users and makes the code harder to maintain. Large React components should be broken into smaller focused components with clearer responsibilities. [web:1062][web:1059]

Success criteria:
- Dashboard is split into role-specific subcomponents
- ProjectDetails is split into overview + tab-level components
- Role-specific hierarchy is clearer
- Client-facing experience feels calmer and more premium
- Desktop and phone layouts remain intact from Phase 1
- Existing logic, data, permissions, and routes continue to work
- Code becomes easier to maintain without changing core behavior

Implementation plan:

1. Refactor Dashboard into role-specific components
Take the current Dashboard.tsx and split it into smaller components, such as:
- AdminDashboardView
- CrewDashboardView
- ClientDashboardView

Keep Dashboard.tsx as the orchestrator/container only.

Dashboard container responsibilities:
- auth + effective role resolution
- top-level shared loading and error states
- shared page shell / page title
- pass required data into role-specific views

Role-specific views:
AdminDashboardView:
- top summary / status cards
- “needs attention” section
- project grid
- quick actions

CrewDashboardView:
- today’s tasks
- upcoming events
- quick access to active projects
- lightweight action flow

ClientDashboardView:
- project summary
- next milestone
- latest update
- important dates
- direct links to relevant project sections

Do not add new business workflows.
Focus on reorganizing existing content into more readable and maintainable role-specific layouts.
Role-based dashboards work best when each role sees only the information and actions most relevant to them. [web:1055][web:760]

2. Improve Dashboard hierarchy
For all role views:
- establish a stronger visual order
- put most important status/action content near the top
- reduce equal-weight sections
- avoid overloading the first viewport
- keep quick actions near the top for admin
- keep crew dashboard action-oriented
- keep client dashboard reassurance-oriented and simpler

Information architecture should help users quickly understand what matters first and what action to take next. [web:1063][web:1066]

3. Refactor ProjectDetails into smaller sections/components
Take the large ProjectDetails page and split it into smaller components.

Target structure:
- ProjectDetails.tsx becomes the page shell/container
- Extract a reusable header section
- Extract a project overview section
- Extract tab components or grouped tab renderers, for example:
  - ProjectOverviewTab
  - ProjectPlanTab
  - ProjectCollaborationTab
  - ProjectVisualsTab
  - ProjectFinancialTab

If the existing app already has specific tab names, preserve the functional behavior, but reorganize the rendering structure.

Do not remove functionality.
Do not remove tabs.
You may regroup or structure rendering logic more cleanly internally.

4. Improve ProjectDetails page hierarchy
Without changing the underlying data, make the page easier to scan.

Preferred order:
- project header
- key summary / health overview
- next important action or pending item
- tabs / sections
- tab content

In the header or overview area, make sure the most important project context is easy to see:
- project name / address
- status
- timeline / dates
- latest meaningful update
- relevant budget visibility status
- key contact or presence signal if already present

The goal is to make ProjectDetails feel like a true project hub, not just a tab strip over a giant page.

5. Improve client-facing ProjectDetails experience
For client users:
- simplify the presentation
- reduce admin/internal noise
- make language more client-friendly where feasible
- prioritize progress, next step, updates, photos/documents, and relevant communication

Do not expose internal-only content to clients.
Do not weaken permissions.
Do not change backend access logic.

6. Improve component boundaries
As you refactor:
- keep business/data fetching logic in containers or hooks where practical
- keep presentational components focused on UI
- avoid adding more inline JSX complexity
- reduce duplication where possible
- prefer smaller readable components over monolithic render blocks

React best practices favor smaller focused components and clearer separation between UI and logic for maintainability. [web:1062][web:1064]

7. Preserve ProjectCard and shared components
Review ProjectCard and related reused UI components.
Only make small safe improvements if needed to support the new hierarchy and consistency.
Do not redesign cards unnecessarily.
Do not create churn just for code style reasons.

8. Mobile and adaptive behavior
Preserve the adaptive device behavior established in Phase 1.
Do not regress:
- mobile dashboard readability
- mobile ProjectDetails tab usability
- Planning Board phone limitations
- dense tool banners on phone if already added

9. Validation and QA
Before finishing, verify:
- Dashboard still works for admin, crew, and client
- role switching still works for admin preview mode
- ProjectDetails still loads all sections and tabs
- client only sees allowed information
- no routes are broken
- mobile and desktop layouts both still work
- no populated data has changed or been removed
- refactoring did not change behavior unintentionally
