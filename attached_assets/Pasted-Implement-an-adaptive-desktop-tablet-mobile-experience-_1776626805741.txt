Implement an adaptive desktop / tablet / mobile experience for the Aster & Spruce Connect app.

Goal:
Do not treat mobile as a squished version of desktop.
Keep the same product, branding, and core workflows, but create device-appropriate layouts and feature behavior for desktop, tablet, and phone.

Critical rules:
- Preserve all existing populated data.
- Do not wipe, reset, overwrite, or migrate data destructively.
- Do not remove existing routes or break existing role guards.
- Do not change API contracts unless absolutely necessary.
- Do not rewrite the app from scratch.
- Make focused, incremental, non-destructive UI changes only.
- Keep all existing desktop functionality working unless a feature must be intentionally limited on smaller screens for usability.

Primary UX decision:
This app needs adaptive behavior, not only generic responsive shrinking.
Desktop should remain the full workspace.
Tablet should be a reduced but still capable workspace.
Phone should be a focused companion experience for quick actions, status checks, communication, uploads, and lightweight review.

Why:
Complex dashboards, dense forms, and canvas-based tools do not always translate well to small touchscreens. Adaptive design should change layout and interaction patterns based on available space and device capability rather than forcing the same interface everywhere. [web:1026][web:1029][web:1037]

Device strategy:

1. Desktop experience
Desktop is the primary workspace for internal/admin use.
Keep or support full functionality for:
- Planning Board / Spatial Canvas
- Cost Estimator
- Payroll
- Supplier Prices
- deep Project Details work
- multi-panel layouts
- dense project management workflows

2. Tablet experience
Tablet should support:
- project review
- planning board viewing
- light board editing if usable
- project updates
- calendar review
- documents, chat, photos, and checklist interactions

Tablet may use reduced-density layouts and simplified toolbars.

3. Phone experience
Phone should be treated as a focused companion experience.
Prioritize:
- dashboard overview
- project status
- today’s tasks
- upcoming events
- chat/messages
- photo upload
- quick checklist updates
- quick time entry
- basic profile/account actions
- project updates and review

Do not force advanced desktop tools to behave the same way on phone.

Planning Board / Spatial Canvas requirements:
This is the most important mobile adaptation.

Current problem:
The Planning Board is not helpful on the phone.

Required change:
Create device-specific behavior for Planning Board.

Desktop:
- full editing remains available

Tablet:
- allow view mode and possibly limited editing if the experience is strong and usable

Phone:
- do NOT force the full Fabric.js editing experience if it is cramped or difficult
- provide a simplified mobile mode, one of:
  - view-only mode
  - limited quick-edit mode
  - comment / note / approve mode
- show a clear message such as:
  “For full board editing, open this Planning Board on desktop or tablet.”
- still allow useful mobile actions such as:
  - browse boards
  - zoom and inspect
  - view linked project context
  - add a note or reaction if feasible
  - upload inspiration/reference if feasible

Design rule:
If a feature requires precision, drag/drop, dense controls, or simultaneous context, optimize it for desktop and simplify it on mobile.

Pages to review and adapt:

- client/src/components/PlanningBoard.tsx

- any shell/layout/sidebar components
- other pages where desktop density may not work on phone

Implementation requirements:

A. Breakpoint strategy
Use content-driven breakpoints and adaptive behavior, not just visual shrinking.
Suggested breakpoints:
- phone
- tablet
- desktop
Use your judgment based on actual content density.

B. Navigation
Desktop:
- sidebar or persistent visible nav
Tablet:
- compact visible nav or hybrid pattern
Phone:
- mobile-friendly drawer, tab bar, or compact menu
Make sure navigation is always easy to discover.

C. Dashboard adaptation
Desktop:
- richer overview with more simultaneous information
Phone:
- simplify into stacked cards
- prioritize “what needs attention now”
- keep primary actions thumb-friendly
- reduce density and avoid wide multi-column layouts

D. Project Details adaptation
Desktop:
- full project hub with richer header and broader tab visibility
Phone:
- simplify the header
- reduce visual noise
- consider grouping content into a smaller set of mobile-friendly sections
- prioritize updates, tasks, chat, checklist, and files over heavy admin detail
- avoid showing too many equal-weight tabs at once

E. Dense tools and tables
For tools like:
- Cost Estimator
- Supplier Prices
- Payroll
- possibly Gantt or timeline-heavy views

On phone:
- either simplify heavily
- convert to stacked cards / key-value layouts
- or clearly recommend desktop/tablet for full use
Do not ship unusable cramped interfaces.

F. Mobile interaction rules
- touch targets must be comfortable
- buttons and controls should be thumb-friendly
- reduce hover-only interactions
- avoid hidden critical actions that rely on precise tapping
- keep critical actions visible and reachable

G. Performance and loading
On smaller devices:
- avoid loading or rendering heavy desktop-only UI where possible
- lazy-load complex components if appropriate
- reduce unnecessary visual overhead

H. Preserve role behavior
Admin, Crew, and Client role behavior must still work correctly across device sizes.
Do not weaken role-based access or visibility.

I. Validation requirements
After implementation, verify:
1. Desktop still supports full internal/admin workflows
2. Phone experience feels intentionally simplified, not broken
3. Planning Board is no longer frustrating on phone
4. Client mobile experience feels polished and easy to use
5. Navigation remains clear on all device sizes
6. No data has been removed or corrupted
7. No existing routes are broken
